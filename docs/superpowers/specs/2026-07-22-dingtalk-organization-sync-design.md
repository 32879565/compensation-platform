# 钉钉组织同步完善设计

日期：2026-07-22  
状态：已批准  
适用项目：Compensation Platform

## 1. 背景

仓库已经有一套未提交的钉钉组织同步实现，覆盖组织快照读取、门店预览与确认、厅面/厨房复核人同步、身份盲索引、并发保护、审计和敏感工资访问时效校验。相关后端定向测试 40 项、前端定向测试 10 项已经通过。

当前实现仍以门店为中心，使用根部门名称发现同步范围，缺少完整区域层级、稳定根部门 ID、定时预览和生产级收尾。本设计在保留现有两阶段安全模型的基础上增量完善这些能力，不重写已经验证的并发和授权机制。

## 2. 已确认的业务决策

1. 钉钉是区域、门店层级及门店复核人路由的权威来源。
2. 同步范围是配置的一个或多个钉钉根部门子树，不读取整个企业通讯录。
3. 同步完整的区域—门店层级，并同步厅面、厨房两类复核人。
4. 钉钉中缺失的本地组织在确认后停用；历史记录不删除。
5. 任一结构或身份冲突都会阻止整批确认；应用必须是单事务、全有或全无。
6. 首次组织匹配使用标准化后的完整相对路径和名称；绑定后只认钉钉部门 ID。
7. 复核人首次按工号精确匹配本地员工；绑定后使用钉钉用户 ID。禁止手机号或姓名兜底。
8. 每日自动生成预览并通知 HR，但不自动应用；HR 确认后才写入正式数据。
9. 发薪复核和敏感工资披露前继续进行实时组织授权校验。
10. 根部门下、门店之前的组织节点识别为区域；名称以“店”结尾、已有门店绑定或与现有门店完整路径精确匹配的节点识别为门店。门店下的厅面、厨房等部门不创建本地组织节点。

## 3. 目标与非目标

### 3.1 目标

- 以稳定远端 ID 同步区域、门店的新增、绑定、改名、移动、启用和停用。
- 安全同步每家门店的厅面、厨房复核人及其最小权限范围。
- 通过预览、冲突阻断、二次校验和原子事务避免部分应用及错误授权。
- 每日发现组织变化并向具备同步权限的 HR 发出提醒。
- 保留完整、可追溯且不泄露钉钉原始身份标识的审计证据。
- 保持手动触发、定时触发和重复确认的幂等性。

### 3.2 非目标

- 不把钉钉同步扩展为通用 HRIS，也不管理劳动合同、招聘或员工档案全量字段。
- 不同步门店内部的普通班组为本地 `OrgUnit`。
- 不从本地反向修改钉钉。
- 不自动应用定时任务发现的变化。
- 不使用姓名、手机号或模糊相似度自动绑定员工或组织。
- 不硬删除组织、员工、工资历史或审计记录。

## 4. 总体架构

采用现有的“读取快照 → 生成持久化预览 → HR 确认 → 原子应用 → 敏感访问时重新校验”架构。

### 4.1 组件边界

1. **钉钉客户端**：负责分页读取指定根部门的部门和成员快照，校验响应结构，并将供应商错误转换为稳定错误。它不访问本地组织表。
2. **节点分类器**：根据根映射、远端路径、门店名称规则和已有稳定绑定，将远端节点分类为区域、门店或门店内部部门。它是无数据库写入的纯逻辑。
3. **差异规划器**：读取本地快照，计算组织与复核人动作、冲突、警告和基线指纹，不修改正式组织或权限数据。
4. **同步批次存储**：持久化远端快照哈希、同步边界、触发来源、过期时间、差异项、匹配依据和状态。
5. **原子应用器**：取得组织同步锁，重新读取远端快照并校验本地基线，在单一数据库事务中应用所有动作。
6. **授权时效守卫**：发薪通知和经理查看敏感工资时，验证最新应用批次、当前门店绑定、当前复核范围及钉钉实时身份。
7. **一次性调度命令**：每日由唯一外部调度实例调用，生成预览并通知 HR。命令本身使用数据库 advisory lock 防止重复运行。
8. **组织页面**：展示最近批次、分类差异、冲突原因、过期状态和确认入口。

各组件通过快照、预览对象和稳定错误码通信。供应商 DTO、匹配逻辑、数据库写入和 HTTP 展示不得相互混合。

## 5. 同步边界与配置

以远端 ID 配置同步边界，替换当前仅按根名称发现范围的生产路径。每个远端根部门必须映射到一个现有的本地非门店锚点，锚点使用不可变本地组织代码标识：

```text
COMP_DINGTALK_ORG_ROOT_MAPPINGS=<remote-root-id>:<local-anchor-code>,...
```

约束如下：

- 远端根 ID 必须为正整数且不得重复。
- 本地锚点必须存在、未删除、处于启用状态，且类型不是 `STORE`。
- 不同远端根不得映射到同一个远端子树；远端子树交叠时启动或预览失败。
- 同一远端节点最多属于一个配置根。
- 生产 `live` 模式启用读取同步时必须配置根映射；旧的根名称配置只用于兼容 sandbox 测试，不能作为生产权威配置。
- 调度时区默认 `Asia/Shanghai`，执行时间默认每日 09:00，均可通过部署配置修改。
- 预览确认窗口沿用 15 分钟。定时预览过期后仍保留为变更证据，但 HR 必须刷新预览后才能确认。

厅面和厨房经理职务名称继续使用互不重叠的精确配置集合。所有名称在比较前执行 Unicode NFKC、首尾空白清理、连续空白折叠和大小写归一化；保存展示值时保留钉钉原始名称。

## 6. 节点分类与匹配

### 6.1 节点分类

对每个配置根的后代按以下顺序分类：

1. 已通过钉钉部门 ID 绑定到本地 `STORE` 的节点是门店。
2. 与现有本地门店的标准化完整相对路径和名称唯一精确匹配的节点是门店。
3. 名称标准化后以“店”结尾的节点是门店。
4. 门店祖先与配置根之间的节点是区域。
5. 门店以下的节点是门店内部部门，不创建 `OrgUnit`，但可用于判断人员归属。
6. 不属于任何门店路径、又不能构成区域祖先的节点忽略并产生可见警告。

以下情况属于结构冲突并阻止整批确认：部门环、缺失父节点、路径深度超过安全上限、同一节点落入多个根、门店嵌套门店、同一路径出现多个远端门店、一个远端节点对应多个本地节点，或多个远端节点对应同一本地节点。

### 6.2 组织匹配

- 已绑定节点只按 `dingtalk_dept_id` 匹配。钉钉改名或移动产生 `UPDATE`，并在变更字段中分别记录 `name`、`parent_id`；不重新按名称绑定。
- 未绑定节点使用“根映射后的本地锚点 + 完整相对路径 + 节点类型”精确匹配。
- 唯一匹配产生 `LINK`；没有匹配产生 `CREATE`；多重匹配产生冲突。
- 历史状态节点再次出现时产生 `ACTIVATE`，同时更新名称和父级。
- 已启用本地区域或门店在权威远端范围内不再出现时产生 `DEACTIVATE`。停用按子节点优先顺序执行，保留远端 ID 和所有历史引用。
- 本地锚点自身不由钉钉改名、移动或停用。

### 6.3 复核人匹配

- 候选人必须处于有效状态，并属于目标门店子树。
- 职务名称必须精确归入厅面或厨房经理集合，不能同时属于两类。
- 首次绑定要求钉钉工号与本地 `Employee.emp_no` 标准化后唯一精确相等。
- 成功后将钉钉用户 ID 加密保存，并保存域隔离盲索引；后续只按该稳定身份绑定。
- 一个门店、一个复核部门最多有一个有效复核人。
- 没有符合条件的经理产生 `REMOVE_SCOPE` 警告动作，用于及时撤销离任者权限，但不视为匹配冲突。
- 多名候选经理、缺失工号、重复工号、远端身份已被其他员工占用、同一经理跨多个门店，均产生冲突并阻止整批确认。

## 7. 预览与原子应用

### 7.1 预览

手动 API 和定时命令使用同一服务生成预览。预览阶段允许写入同步批次及差异项，但不得修改 `OrgUnit`、员工身份绑定、用户角色或复核范围。

每个批次保存：

- 公共批次 ID、租户、触发来源（`MANUAL` 或 `SCHEDULED`）、创建人与时间；
- 配置根映射的规范化哈希；
- 远端组织快照哈希和本地匹配基线哈希；
- 区域、门店、复核人三类差异项；
- 每项的动作、状态、匹配方法、远端加密身份或盲索引、目标本地 ID、应用前指纹；
- 新增、更新、停用、复核人变更、冲突和警告汇总计数；
- 过期时间、应用人、应用时间和失败码。

相同根配置、相同远端快照和相同本地基线的重复定时执行不创建重复待处理批次；它更新“最后检查时间”并保持幂等。手动刷新可以创建新的可确认批次，同时将同范围旧的未应用批次标记为过期。

### 7.2 确认前校验

确认操作要求调用者具有全局组织同步权限。应用器必须：

1. 获取全局组织同步 advisory lock 和相关授权表锁。
2. 锁定目标批次及全部差异项。
3. 验证批次处于可确认状态、未过期且冲突数为零。
4. 重新读取同一根范围的钉钉快照并比较快照哈希。
5. 重新读取本地组织、员工、用户、角色和复核范围并比较每项基线。
6. 任一验证失败时不写正式数据，将批次标记为过期或失效，并要求重新预览。

### 7.3 应用顺序

全部动作在一个数据库事务中执行：

1. 创建、绑定和启用区域，父级优先。
2. 移动、改名区域。
3. 创建、绑定、启用、移动和改名门店。
4. 绑定复核人员工身份，创建或恢复最小化经理账号和角色。
5. 替换每个门店、部门的唯一复核范围。
6. 撤销不再有效的复核范围、相关授权证明和登录会话。
7. 停用远端缺失的门店和区域，子级优先。
8. 写入逐项应用结果和单条汇总审计记录，提交事务。

数据库唯一约束、并发指纹或供应商快照校验失败时整批回滚。已经成功应用的批次重复确认返回原结果，不重复执行写操作。

## 8. 数据模型与迁移

在现有未提交的 D20 迁移基础上增量调整，避免形成两个相互依赖但尚未发布的迁移：

- `OrgUnit.dingtalk_dept_id` 用于区域和门店，继续保持“非空时全局唯一、必须为正整数”。
- 同步差异项类型增加 `REGION`，保留 `STORE` 和 `REVIEWER`。
- 增加显式动作枚举，至少包含 `LINK`、`CREATE`、`UPDATE`、`ACTIVATE`、`DEACTIVATE`、`ASSIGN_SCOPE`、`REMOVE_SCOPE` 和 `NO_CHANGE`；差异项另存变更字段集合，使一次 `UPDATE` 可以同时表达改名和移动。匹配方法不再承担动作编码职责。
- 批次增加触发来源、根配置哈希、最后检查时间和分类汇总计数。
- 差异项保存父级目标、节点类型及应用前基线指纹。
- 新增独立的组织同步通知投递表，保存同步批次、收件人、状态、尝试次数、稳定错误码、供应商任务 ID 和幂等键。现有工资投递表的工资批次、门店和部门字段继续保持非空，不为组织通知放宽约束。
- 远端用户 ID 保持加密，匹配与唯一性使用带域隔离的盲索引；API、日志和审计不得返回原始 ID。
- 迁移继续在 DDL 前检查重复复核范围和冲突身份，任何歧义均 fail-closed。

迁移必须支持 PostgreSQL fresh upgrade、已有数据库 upgrade 和 downgrade 测试。由于 D20 尚未进入已发布主线，允许在提交前修正其 schema；一旦发布则只能新增后续迁移。

## 9. API 与界面

保留现有接口并扩展响应：

- `POST /api/dingtalk/sync/organization/preview`：生成新的手动预览。
- `GET /api/dingtalk/sync/organization/latest`：读取当前用户可见的最近批次及状态，不访问钉钉。
- `POST /api/dingtalk/sync/organization/{batch_id}/apply`：沿用当前确认接口；路径中的批次公共 ID 必须符合 32 位十六进制格式，请求体不得覆盖任何预览内容。

预览响应按区域、门店、复核人分组，统一返回动作、状态、远端路径、目标本地路径、冲突码和不含 PII 的说明。前端提供：

- 最近检查时间、触发来源、过期状态和变化汇总；
- 区域、门店、复核人三个差异表；
- 根据动作及变更字段展示新增、移动、改名、启用、停用、授权、撤权和冲突标签；
- 仅在批次未过期且冲突数为零时启用确认按钮；
- 应用成功但组织树刷新失败时明确提示“应用已成功”，避免重复确认；
- 供应商故障、批次过期和并发变化对应可操作的“重试读取”或“重新预览”提示。

具有全局同步权限的 HR 才能生成或确认批次。普通组织读取权限只能查看已经应用的正式组织树，不能读取钉钉预览身份信息。

## 10. 定时预览与通知

提供一个一次性 CLI 入口，由部署环境每天调用一次。选择外部唯一调度而不是 FastAPI 进程内定时器，避免多 worker 重复执行。任务流程：

1. 获取非阻塞调度 advisory lock；已有任务运行时安全退出。
2. 读取配置根并生成或复用幂等预览。
3. 无变化时只更新健康指标和最后检查时间。
4. 有变化或冲突时，向所有具备全局组织同步权限、状态有效且已绑定钉钉身份的 HR 发送摘要通知。
5. 通知只包含变化数量、冲突数量和组织页面链接，不包含员工姓名、工号或工资数据。
6. 通知投递使用独立的组织同步 delivery 表，并沿用现有投递状态、幂等键、有限重试和结果审计语义；通知失败不撤销已成功生成的预览。

任务暴露最近成功时间、失败时间、失败码、变更数和冲突数指标。超过 26 小时没有成功检查时触发运维告警；发薪敏感操作仍由现有时效守卫独立 fail-closed，不能依赖定时任务成功来放宽授权。

## 11. 错误处理与安全

供应商超时、限流和 5xx 使用带抖动的指数退避，单次操作最多尝试 3 次；无效凭据、权限不足、结构错误和业务冲突不重试。稳定错误码至少覆盖：

- `ORG_ROOT_CONFIG_INVALID`
- `ORG_ROOT_NOT_FOUND`
- `ORG_SNAPSHOT_INVALID`
- `ORG_NODE_CLASSIFICATION_CONFLICT`
- `ORG_PATH_AMBIGUOUS`
- `ORG_MANAGER_AMBIGUOUS`
- `ORG_EMPLOYEE_MATCH_FAILED`
- `ORG_IDENTITY_CONFLICT`
- `ORG_PREVIEW_HAS_CONFLICTS`
- `ORG_PREVIEW_EXPIRED`
- `ORG_REMOTE_SNAPSHOT_CHANGED`
- `ORG_LOCAL_BASELINE_CHANGED`
- `ORG_CONCURRENT_CHANGE`
- `ORG_PROVIDER_UNAVAILABLE`

HTTP 层将这些错误映射为一致状态码和中文用户提示，但日志保留稳定错误码。所有失败默认拒绝写入或披露，不能静默跳过冲突项。

安全要求：

- 钉钉凭据只从 secret 配置读取，不进入数据库预览、日志或前端。
- 原始钉钉用户 ID 仅加密保存，不出现在 API、审计明细或通知。
- 组织预览不包含工资数据。
- 所有确认、撤权、停用和敏感读取继续写入追加式审计。
- 组织或复核人手工变更必须使对应的同步授权证明失效。
- 停用或替换复核人时撤销其活动会话，避免权限缓存继续有效。

## 12. 测试与质量门禁

### 12.1 后端

- 节点分类：多根、区域、门店、内部部门、嵌套门店、环、孤儿节点和深度上限。
- 组织匹配：首次完整路径匹配、稳定 ID、创建、改名、移动、启用、停用和歧义。
- 复核人：工号精确匹配、稳定身份、零/一/多候选、跨店、身份冲突和撤权。
- 批次：无写预览、TTL、幂等定时运行、重复确认、快照变化和本地基线变化。
- 原子性：应用中任一步骤失败均无正式组织或权限部分写入。
- 并发：预览/应用、手工组织修改、用户绑定修改和经理访问之间的锁行为。
- 权限与审计：全局权限、组织范围、敏感字段脱敏和完整审计。
- 调度：唯一锁、重试上限、通知幂等、通知失败隔离和健康指标。
- PostgreSQL 迁移：fresh upgrade、旧数据回填冲突、upgrade/downgrade 和单一 head。

### 12.2 前端与端到端

- 三类差异展示及各动作标签。
- 冲突、过期、供应商失败和并发变化时的按钮状态与提示。
- 成功应用后组织树刷新，以及刷新失败时不误导用户重复应用。
- 浏览器端完整流程：生成预览、阻断冲突、解决外部数据后刷新、确认、查看更新组织树。

### 12.3 必须通过的门禁

- 修复当前 6 个 Ruff 格式/导入错误。
- 后端 Ruff、Black、Mypy、完整 pytest 与覆盖率门槛通过。
- 前端 ESLint、TypeScript、Vitest 和生产构建通过。
- 组织同步 PostgreSQL 测试及关键 Playwright 流程通过。
- `git diff --check` 无空白错误，Alembic 只有一个 head。

## 13. 上线与回滚

1. 在 sandbox 配置真实结构的脱敏镜像，验证根映射和节点分类。
2. 在隔离 UAT 库执行只读预览，由 HR 对区域、门店及两类复核人逐项签字。
3. 备份 UAT 数据库，完成首次确认，并核对组织数量、停用清单、复核范围和审计。
4. 在生产配置真实根 ID、凭据、公开链接和唯一调度任务；首次同步只允许手动预览与确认。
5. 首次生产确认验收后再启用每日调度。
6. 上线后监控最近成功检查时间、冲突数量、通知投递和敏感访问时效拒绝率。

应用前失败通过事务自动回滚。已成功确认的组织变更不通过数据库迁移倒退；业务误操作使用下一次权威钉钉快照生成补偿同步。灾难性故障按运维手册恢复应用前数据库备份，并重新生成预览。代码回滚必须保留 D20 schema，除非确认没有任何 D20 数据且执行了受控数据库恢复。

## 14. 验收标准

实现满足以下条件时视为完成：

1. 配置根下的区域和门店能够新增、绑定、改名、移动、启用和停用，结果与钉钉权威树一致。
2. 首次匹配只接受完整路径和工号的唯一精确匹配；所有歧义均可见且阻止确认。
3. 任一冲突、过期、远端变化、本地并发变化或数据库失败都不会留下部分正式写入。
4. 钉钉缺失门店被停用，历史工资仍可按权限查询，旧复核人权限和会话立即失效。
5. 每家有经理的门店均准确建立厅面、厨房唯一复核范围；无经理时旧范围被撤销。
6. 每日预览幂等生成，有变化或冲突时 HR 收到不含 PII 的通知。
7. 发薪通知和经理敏感查看仍要求当前钉钉组织身份与最近应用批次一致。
8. 全部质量门禁通过，迁移单一 head，运维文档包含配置、调度、故障处理和首次上线步骤。
