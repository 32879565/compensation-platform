# 蓝图完成审计与交付结论

更新时间：2026-07-21。权威需求仍为 `plans/compensation-platform.md` 与
`docs/payroll-batch-spec.md`；本文记录当前仓库的可复验证据和生产前置条件，不替代业务规则。

## 结论

S1–S17 的应用代码、数据库迁移、自动化测试、容器部署配置和运维脚本已形成可运行闭环。
一次性 Docker 环境已真实跑通：登录 → 主数据 → 考勤 → 核算 → 门店复核 → HR 终审 →
锁定 → XLSX 导出 → 员工工资单查询。

该结论表示“工程交付可进入 UAT/生产准备”，不表示已经替业务方完成法规确认、历史数据
迁移、真实钉钉开通、银行模板认证或生产基础设施操作。下文所列外部闸门完成前，不应宣称
正式生产上线或真实资金发放已就绪。

## 生产发布前必须完成的外部闸门

1. 由薪酬/法务确认广州、深圳、东莞、珠海的费率、基数、生效期、专项扣除来源、兼职/离职
   口径、审批路由、数据保留和销毁政策；将批准版本录入并用官方样例对账。
2. 提供旧系统 138MB 数据及对账基线，确定允许差异和无工号认领负责人；在隔离库实跑迁移，
   签署行数、总额、抽样明细和差异报告。
3. 提供社保/个税官方申报模板与开户行代发模板，完成字段映射、样例验证和银行/经办机构
   验收；现有文件只用于通用对账。
4. 提供钉钉企业应用凭据、正式收件人路由、申诉时限和 H5/回系统决策，并实施/验证真实
   token 与工作通知传输；当前沙盒绝不会对外发送。
5. 批准历史期申诉的红冲、补差和资金结算政策；现有系统只创建受控工作项，不自动移动资金。
6. 在生产环境配置独立 secret manager、DNS/TLS、监控告警和受控备份目录，执行并记录一次
   恢复演练、PII 解密抽样和回滚决策演练。
7. 在目标提交上取得 GitHub Actions 全绿记录，特别是 Ubuntu 严格 `pip-audit`、镜像构建、
   Caddy/Compose 校验和一次性全链路 E2E。

以上闸门有真实输入后，应按 `docs/operations.md` 执行 UAT 与上线；不得用示例费率、示例凭据
或通用对账文件替代正式业务批准。
